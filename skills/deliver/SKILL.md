---
name: deliver
description: |
  Use this skill to deliver a ready-to-send text
  artifact produced by another plugin, skill, or agent
  to a Telegram chat without rewriting, summarizing, or
  reformatting it: read the bot token from the
  `WITNESS_TOKEN` environment variable and the chat id
  from the `WITNESS_CHAT` environment variable, and post
  the supplied text once via the Telegram Bot API. One
  delivery per run — then stop.
---

Stop the run immediately when either `WITNESS_TOKEN` or
  `WITNESS_CHAT` is unset or empty: do not call the
  Telegram Bot API, do not retry, and report a
  single-line failure naming the missing variable.

Read the bot token from the `WITNESS_TOKEN` environment
  variable and the chat id from the `WITNESS_CHAT`
  environment variable, treat `WITNESS_CHAT` as the
  integer chat identifier expected by the Telegram Bot
  API, and use those values for every Telegram Bot API
  call in this run.

Take the message body verbatim from the text artifact
  handed to this skill by the previous plugin, skill, or
  agent — the exact characters of the supplied string,
  whitespace, line breaks, and Markdown intact — and
  send it as the Telegram message body.

Never rewrite, summarize, paraphrase, shorten, reorder,
  translate, prepend, append, or otherwise alter the
  supplied text; the only permitted transformation is
  the `MarkdownV2` escaping required by Telegram and
  defined in the rule below.

Stop the run immediately when the supplied text
  artifact is missing, empty, or only whitespace: do not
  call the Telegram Bot API, do not invent a body, and
  report a single-line failure naming the missing
  artifact.

Set `parse_mode` to `MarkdownV2` on the API call and
  escape the Telegram reserved characters `_`, `*`, `[`,
  `]`, `(`, `)`, `~`, `` ` ``, `>`, `#`, `+`, `-`, `=`,
  `|`, `{`, `}`, `.`, `!` with a leading backslash
  everywhere outside of inline code spans, link labels,
  and link URLs, because an unescaped reserved character
  makes Telegram reject the message with HTTP 400.

Stop the run immediately when the supplied text exceeds
  four thousand characters, because Telegram caps a
  single `sendMessage` payload at four thousand and
  ninety-six characters and a longer text is rejected
  outright rather than truncated; report a single-line
  failure naming the actual length and the cap.

Post the message once with
  `curl -sS -X POST
  "https://api.telegram.org/bot$WITNESS_TOKEN/sendMessage"
  -d chat_id="$WITNESS_CHAT"
  -d parse_mode=MarkdownV2
  -d disable_web_page_preview=true
  --data-urlencode text@-`
  and feed the supplied message body to that command
  through standard input, so backticks, parentheses,
  newlines, and reserved characters survive the shell
  intact and do not need to be re-escaped on the
  command line.

Pass `disable_web_page_preview=true` on every call so
  Telegram does not expand any URL inside the supplied
  text into a link-preview card under the message; the
  preview card would add noise the upstream artifact did
  not include.

Read the JSON response on the first POST and verify
  that the top-level `ok` field is `true`; when it is
  `false`, surface the `description` field to the user
  and stop the run, because a blind retry on an unparsed
  error would mask the cause and could send the wrong
  message twice.

Never name Claude, Claude Code, Anthropic, an AI agent,
  a language model, or any other automation tool in the
  message text, and never add a `generated by` line, a
  marketing footer, or any AI-disclosure trailer; the
  delivery preserves the upstream artifact, not the tool
  that transmitted it.

Never log, print, echo, or write the value of
  `WITNESS_TOKEN` to the conversation, to a file, to a
  shell trace, or to a debug line; the token is a
  credential and a leaked credential lets a third party
  impersonate the bot in every chat it has joined.

Stop after the single `sendMessage` call returns
  `ok: true`: do not send a follow-up message, do not
  edit the message, do not pin it, do not start a second
  run, and re-run this skill from the top when the next
  artifact needs its own delivery.
